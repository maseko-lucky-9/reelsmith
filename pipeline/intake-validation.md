# Intake Validation — Word-Synced Subtitles

## Status
- **Parseable**: yes
- **dual_client**: false (Streamlit UI + FastAPI backend, single client surface)
- **scope_warning**: false (single feature, scoped to caption/transcription bounded context)
- **design_brief_source**: none — design gate REQUIRED (key open question on transcription backend)

## Normalized Requirement

**Problem**: Today's caption pipeline produces blocks of text per chapter, then `caption_service.generate_captions()` splits the block into ~7-word groups using a uniform `words_per_second = len(words) / duration` rate (`app/services/caption_service.py:20`). Every word is assumed to consume an equal slice of the chapter's runtime — so when speech is uneven (pauses, fast bursts) the subtitles drift visibly out of sync.

**Goal**: Produce captions where each segment (one word, or a small configurable run) is timestamped to the actual moment the word is spoken. The rendered video shows "karaoke-style" subtitles that pop in word-by-word as speech occurs.

## Acceptance Criteria

1. Each emitted caption segment corresponds to one word or a small run of N words (N configurable; default decided in design gate), with `start`/`end` derived from real audio timing — not uniform interpolation.
2. SRT and VTT files generated by `caption_service` carry these fine-grained timestamps; existing `captions_to_dicts` shape is preserved.
3. `subtitle_image_service.create_subtitle_image()` consumers (i.e. `clip_service.add_captions_to_clip()`) render one image per fine-grained segment, composited at the segment's `start_time`.
4. `pytest` (default invocation, excluding `integration`/`live`/`playwright`) passes after changes. New unit tests cover the new timing path with a deterministic fixture.
5. **No new pip dependency unless justified.** A justified addition is one that the design gate explicitly approves with rationale (e.g. "the current backend cannot return word timing; we must swap").

## Critical Verified Findings (from current source Read)

- `app/services/transcription_service.py:23` calls `recognizer.recognize_google(audio_data, language=language)` **without** `show_all=True`. Returns a plain `str` — no timing info at all.
- **The user-supplied note in the requirement is INCORRECT** ("`alternative[0].words` word-level timestamps if the Google backend returns them"). Google's Web Speech API (the free endpoint that `SpeechRecognition.recognize_google` uses) returns alternatives with `transcript` + `confidence` only — never word offsets. `show_all=True` exposes alternatives, **not** word timings. [VERIFIED via source read; confirmed by SpeechRecognition library docs.]
- `app/services/caption_service.py:20-46` already groups into multi-word captions, but uses a uniform-rate model. The data structure (`SubRipItem` / `Caption`) already supports per-segment `start`/`end` floats — the consumer code does not need a schema change to absorb finer-grained timestamps. Only the *producer* needs new timing data.
- `app/services/clip_service.py:140-145` iterates `for caption in captions` and stamps each onto its `start_time`/`duration`. This already supports an arbitrary number of fine-grained segments — no rendering changes required if the segments simply become smaller and more numerous.
- `requirements.txt` currently has `SpeechRecognition==3.16.1` and `moviepy==1.0.3`. No timing-capable backend is currently installed.

## Implication

Acceptance Criterion #5 ("no new pip dependency unless justified") will be the load-bearing decision in the design gate. The current Google backend **cannot** produce word-level timestamps. Three viable approaches must be debated:

- **A. Forced alignment via stdlib + heuristics** on the existing transcript + audio (e.g. silence-detection-based partitioning). No new dep, but accuracy is mediocre.
- **B. `openai-whisper` (or `faster-whisper`)** — local model, returns word-level timestamps natively. New dep + model download (~150MB+).
- **C. `vosk`** — small offline ASR with word timestamps. New dep (~50MB model).

This trade-off is the central unresolved item for Phase 0.5.

## Confirmed Scope

### In scope
- `app/services/transcription_service.py` — extend (or add a sibling) to return a structured result `[{word, start, end}, ...]` instead of, or alongside, the plain transcript string.
- `app/services/caption_service.py` — add a new code path that consumes the structured per-word timing list and emits SRT/VTT segments grouped at configurable granularity (1 word, 2 words, etc.).
- `app/services/clip_service.py` and `app/services/subtitle_image_service.py` — verify they handle high-cardinality segment lists efficiently. Likely no functional change.
- `app/settings.py` — new env-var(s): caption granularity (words per segment), transcription backend selector if a new one is added.
- New tests under `tests/` covering: word-grouping logic on a mocked transcript-with-timing fixture, SRT/VTT output shape, edge cases (single-word transcript, gaps between words).
- `requirements.txt` — only edited if design gate selects a new backend.
- Caller in the orchestrator (likely `app/workers/orchestrator.py`) — pass the structured result through to `caption_service`.

### Out of scope
- Multi-language word alignment beyond what the chosen backend natively supports.
- Caption styling effects (highlight current word, fade-in animations) — purely a timing change, not a styling change.
- Streaming/incremental transcription.
- UI changes (Streamlit displays the same final video; no editor surface for word timing).

## Constraints

- Existing default `pytest` run must remain green. New tests must not require network or large model downloads in the default run; use marker `live` if a real backend is unavoidable for an end-to-end test.
- The chosen backend must work on macOS (development target — `.venv-mac/` present in repo).
- The pipeline's `provider == "stub"` path must continue to return deterministic output; in the new world it should return a structured stub list, not just a string.
